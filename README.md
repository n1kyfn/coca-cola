# Учебный проект: SPA «Коворкинг» (подготовка к экзамену по React)

**Главный упор в задании — серверная пагинация:** несколько экранов должны ходить в API с параметрами **`page`** и **`limit`**, разбирать ответ **`{ data, total, page, limit }`**, показывать номер страницы, переход «назад / вперёд» (или аналог) и **общее число записей** (`total`). Бэкенд уже отдаёт данные в этом формате — ваша задача корректно связать UI, запросы и при необходимости URL.

Дополнительно отрабатываются **guard и роли**, **loader и toast**, **адаптив**, **валидация форм**, **фильтрация и сортировка каталога**, **карточка зоны по `:id`**, **полнота полей из API**.

Бэкенд — в каталоге [`back/`](back/). Фронтенд вы создаёте самостоятельно (например `front/` или отдельный репозиторий).

## Запуск API

```bash
cd back
cp .env.example .env
npm install
npm run migrate
npm run seed
npm run dev
```

По умолчанию сервер: **`http://localhost:3010`**.

Проверка:

```bash
curl -s http://localhost:3010/health
```

Заголовок авторизации для защищённых запросов: `Authorization: Bearer <accessToken>`.

## Роли и доступы (концепция для фронта)

| Роль | Что делает |
|------|------------|
| **guest** | Просмотр главной и каталога зон; вход/регистрация. |
| **client** | Бронирование, «Мои брони», отзывы (один отзыв на зону). |
| **manager** | Подтверждение/отклонение броней, модерация отзывов, CRUD зон. |
| **admin** | Список пользователей с пагинацией, включение/выключение учётки (`PATCH /users/:id`, поле `isActive`). Брони и статусы — как у manager; модерация отзывов (`GET /reviews/manage`) — как у manager. |

Меню и маршруты должны **зависеть от роли**. Сервер вернёт **403**, если роль не подходит — обработайте это на клиенте (toast / редирект).

## Пагинация и текущий бэкенд

Один контракт для **списков со страницами** (всё в query, заголовок Bearer только где указано):

```json
{ "data": [ /* записи */ ], "total": 42, "page": 1, "limit": 10 }
```

| Экран / данные | Запрос | Замечания |
|----------------|--------|-----------|
| Каталог зон | `GET /spaces?page=&limit=` + фильтры (`zoneType`, `minPrice`, `maxPrice`, `minCapacity`, `sort`) | Источник истины для списка зон с пагинацией. |
| Популярные зоны (виджет) | `GET /spaces/popular?limit=` | Ответ — **массив**, не объект с `data`/`total`; только **`limit`** (на бэке по умолчанию 3, максимум 20). Отдельно от основной пагинации каталога. |
| Мои / все брони | `GET /bookings?page=&limit=` + опционально `status=` | **Bearer**; client/guest видят свои, manager/admin — все. |
| Пользователи (админка) | `GET /users?page=&limit=` | **Bearer**, только **admin**. |
| Отзывы на зону (публично) | `GET /reviews?spaceId=&page=&limit=` | Без скрытых отзывов (`isHidden`). |
| Модерация отзывов | `GET /reviews/manage?spaceId=&page=&limit=` | **Bearer**, **manager** или **admin**; все отзывы зоны. |

На бэке по умолчанию **`page = 1`**, **`limit = 10`**, верхняя граница **`limit` — 100**. Число страниц на клиенте: `Math.ceil(total / limit)` (при `total === 0` договоритесь, как показывать пустой список и кнопки).

**Что проверяется в первую очередь:** запрос уходит с актуальными `page`/`limit`; в UI видны `total` и текущая страница; при смене страницы не теряются **фильтры каталога** (`/spaces`); при переключении страниц есть **loader** или блокировка кнопок (**`isFetching`**), чтобы не дублировать запросы. Подробнее и пример с React Query — [`docs/04-pagination.md`](docs/04-pagination.md).

## Обязательный функционал SPA (чеклист)

### 1. Guard (маршруты и меню)

- Защищённые страницы доступны только с валидным токеном; при отсутствии токена — редирект на страницу входа (или показ общедоступных страниц).
- Страницы только для **manager** / **admin** / **client** не должны попадать в меню других ролей (и желательно не открываться по прямой ссылке без прав).
- При **401** (просроченный токен) — logout / обновление токена через `/auth/refresh` / повторный вход; покажите сообщение (toast).
- При **403** — сообщение пользователю, без «белого экрана».

### 2. Loader

- Индикатор загрузки при запросах: **все экраны со списками и пагинацией** (каталог, брони, пользователи, отзывы), карточка зоны, формы бронирования и входа.
- Допустимы глобальный оверлей или локальные скелетоны; кнопки отправки формы блокируются на время запроса.

### 3. Toast (уведомления)

Покройте toast-сценарии минимум:

- Успешный вход / регистрация / выход.
- Успешное создание брони, смена статуса брони, отмена брони.
- Успешное создание/изменение зоны (manager), удаление зоны.
- Отзыв создан / скрыт / удалён (manager или admin).
- Профиль сохранён (`PATCH /users/me`).
- Админ: изменён флаг `isActive` у пользователя.
- Любая ошибка API (показать `error` из тела ответа при наличии); сетевые ошибки.

### 4. Валидация форм и подсветка ошибок

- **React Hook Form** + **zod** или **yup** + `@hookform/resolvers`.
- У каждого поля: **label** и **placeholder** (где уместно).
- Ошибки: текст под полем + **красная рамка** у инпута; используйте `aria-invalid` для доступности.
- При ответе сервера `400` с `{ "error": "..." }` покажите сообщение (toast и/или общая ошибка формы).

### 5. Вёрстка и медиа

- Единый визуальный стиль (типографика, отступы, цвета).
- На **каждой** странице заголовок уровня **`h1`**.
- **Адаптив**: `@media` или контейнерные запросы; каталог на узком экране не «ломается» (карточки в колонку, читаемые кнопки).

### 6. Пагинация (см. раздел «Пагинация и текущий бэкенд» выше)

- Реализуйте пагинацию минимум на: **каталог зон**, **список броней**, **список пользователей (admin)**; на **странице зоны** — пагинация **отзывов** (публичный список и, для manager/admin, режим модерации с тем же контрактом, другой URL).
- Не подменяйте ответ массивом там, где API отдаёт объект с `data`/`total` (типичная ошибка — перепутать с ответом `GET /spaces/popular`, где как раз массив).

### 7. Фильтрация и сортировка

- Параметры каталога на бэке: `zoneType`, `minPrice`, `maxPrice`, `minCapacity`, `sort` (`price_asc`, `price_desc`, `rating_desc`, `id_asc`), плюс `page`, `limit`.
- Фильтры и сортировка должны **сохраняться при смене страницы** (не сбрасываться без действия пользователя).
- **Дополнительно:** для **одного** из второстепенных списков на клиенте (например избранное на главной без отдельного API) допускается фильтрация **только на фронте** через `useMemo` — цель: понимать разницу между серверной и клиентской фильтрацией (см. `docs/05-filtering-and-sorting.md`).

### 8. Страница зоны по `:id`

- Маршрут вида `/spaces/:id` (или аналогичный).
- Загрузка **`GET /spaces/:id`**, отображение **всех полей** из JSON (включая `images`, `rating`, `description`, числовые поля).
- Обработка **404** с понятным UI.

### 9. Полнота данных

- В таблицах и карточках выводите **все релевантные поля** из ответа API (включая вложенные `Space` / `User` в бронированиях, даты в ISO, статусы).
- Перечитайте это задание и README бэкенда: не «обрезайте» данные без причины.

## Docker для фронтенда (обязательно в рамках проекта)

1. **Development:** `Dockerfile` (или `Dockerfile.dev`), который поднимает **React dev server** (Vite или CRA), порт **проброшен** наружу (`-p 5173:5173` или как у вас в шаблоне). Контейнер должен стартовать **без установленного локально Node** (только Docker).
2. **Production:** **multi-stage build**: стадия сборки (`npm run build`), стадия со **статикой** — например **nginx**, раздаёт содержимое `dist` / `build`. Пример конфигурации и порядок команд — в [`docs/07-docker.md`](docs/07-docker.md).

Переменная **`VITE_*` / `REACT_APP_*`** для базового URL API задавайте на этапе сборки или через env-файл для контейнера.

RTK / RTK Query, TanStack Query, axios, zod, yup, react-hook-form + resolvers, react-router (v6).

## Краткий справочник API

| Метод | Путь | Кто |
|--------|------|-----|
| GET | `/health` | все |
| POST | `/auth/register`, `/auth/login`, `/auth/refresh`, `/auth/logout` | см. роуты |
| GET/PATCH | `/users/me` | Bearer |
| GET | `/users?page&limit` | **admin** |
| PATCH | `/users/:id` body `{ isActive }` | **admin** |
| GET | `/spaces/popular?limit=` | все; ответ — **массив** (не `{ data, total, … }`) |
| GET | `/spaces?page&limit&...filters` | все; ответ **с пагинацией** `{ data, total, page, limit }` |
| GET | `/spaces/:id` | все |
| POST/PUT/DELETE | `/spaces`, `/spaces/:id` | **manager** |
| GET | `/bookings?page&limit&status=` | client/guest — свои; manager/admin — все |
| POST | `/bookings` | client, manager |
| PATCH | `/bookings/:id/cancel` | владелец или manager/admin |
| PATCH | `/bookings/:id/status` | manager, admin |
| GET | `/reviews?spaceId=&page=&limit=` | все (публичные, без скрытых); ответ с пагинацией |
| GET | `/reviews/manage?spaceId=&page=&limit=` | **manager**, **admin** (Bearer) |
| POST/PATCH/DELETE | `/reviews`… | см. реализацию в `back/src/routes/reviews.js` |

Подробные методички: каталог [`docs/`](docs/).
